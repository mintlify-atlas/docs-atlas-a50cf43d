Dynamic components extend the styled component model to support styles that change based on props. Instead of passing a CSS string directly to `%styled.tag`, you pass a function that receives labelled arguments and returns a CSS string. The function arguments become React props on the generated component, with their types inferred automatically from how each argument is used inside the CSS.

## Defining a Dynamic Component

Replace the CSS string with a function that takes labelled arguments and returns a template string containing interpolations:

<CodeGroup>
```rescript ReScript
module Dynamic = %styled.div((~color, ~background) => `
  color: $(color);
  background-color: $(background);
`)

<Dynamic color=CSS.hex("#EB5757") background=CSS.hex("#516CF0")>
  {React.string("Hello!")}
</Dynamic>
```

```reason Reason
module Dynamic = [%styled.div (~color, ~background) => {|
  color: $(color);
  background-color: $(background);
|}];

<Dynamic color={CSS.hex("#EB5757")} background={CSS.hex("#516CF0")}>
  {React.string("Hello!")}
</Dynamic>;
```
</CodeGroup>

The component `Dynamic` now accepts `~color` and `~background` as required props in addition to all of the standard `div` element props.

## Type Holes and Type Inference

styled-ppx uses a technique called **type holes** to infer the types of your function arguments automatically. The PPX inspects the position of each `$(ident)` interpolation inside the CSS string and determines which CSS type is required at that position.

In the example above:
- `color: $(color)` — `color` is inferred as `CSS.Color.t`
- `background-color: $(background)` — `background` is inferred as `CSS.Color.t`

This means you pass values from the `CSS` module and the compiler enforces the correct type for you. If you pass a `CSS.Length.t` where a `CSS.Color.t` is expected, you get a compile error.

<Tip>
  You can read more about the type holes technique in [this Twitter thread by the author](https://twitter.com/davesnx/status/1552305210558742528). The core idea is that the PPX generates a type annotation derived from the property context, and the OCaml type checker does the rest.
</Tip>

## A More Complete Example

Dynamic components work naturally with pattern matching, conditional logic, and any OCaml/ReScript expression—as long as the final returned value is an inline string literal or an inline array of `Css.rule` values (not a variable reference or a named function reference):

<CodeGroup>
```rescript ReScript
type size = Small | Medium | Large

let sizeToPx = size =>
  switch size {
  | Small => CSS.px(8)
  | Medium => CSS.px(16)
  | Large => CSS.px(24)
  }

module Button = %styled.button((~variant, ~size) => {
  let padding = sizeToPx(size)
  let bg = switch variant {
  | `Primary => CSS.hex("516CF0")
  | `Secondary => CSS.hex("EEEEEE")
  }
  `
    padding: $(padding);
    background-color: $(bg);
    border: none;
    border-radius: 4px;
  `
})
```

```reason Reason
type size = Small | Medium | Large;

let sizeToPx = size =>
  switch (size) {
  | Small => CSS.px(8)
  | Medium => CSS.px(16)
  | Large => CSS.px(24)
  };

module Button = [%styled.button (~variant, ~size) => {
  let padding = sizeToPx(size);
  let bg = switch (variant) {
  | `Primary => CSS.hex("516CF0")
  | `Secondary => CSS.hex("EEEEEE")
  };
  {|
    padding: $(padding);
    background-color: $(bg);
    border: none;
    border-radius: 4px;
  |}
}];
```
</CodeGroup>

## All Standard Props Are Still Available

Dynamic component props are merged with the standard HTML element props. The component in the example above accepts `~color`, `~background`, and also `~onClick`, `~className`, `~id`, and any other `div` prop—exactly like a static styled component.

<CodeGroup>
```rescript ReScript
<Dynamic
  color=CSS.hex("#EB5757")
  background=CSS.hex("#516CF0")
  onClick={_ => Js.log("clicked")}
  className="extra"
>
  {React.string("Hello!")}
</Dynamic>
```

```reason Reason

  background={CSS.hex("#516CF0")}
  onClick={_ => Js.log("clicked")}
  className="extra"
>
  {React.string("Hello!")}
</Dynamic>;
```
</CodeGroup>

## Name Collision Warning

If you choose a prop name that collides with one of the standard `makeProps` keys generated by rescript-react, styled-ppx will **override** the standard prop and replace it entirely. The custom prop takes precedence.

<Warning>
  Avoid naming dynamic props after existing element props. For example, using `~size` as a prop name for a `div` component will override the standard `size` DOM attribute. Prefer domain-specific names like `~buttonSize` or `~iconSize` instead.
</Warning>

You can check the list of standard `makeProps` keys in [rescript-react's `ReactDOM.res`](https://github.com/rescript-lang/rescript-react/blob/master/src/ReactDOM.res#L367) to see which names to avoid.

## PPX Limitation: The Return Value Must Be an Inline Literal

Because styled-ppx operates on the AST before compilation, it must be able to read the CSS at transform time. The last expression of the function body (the returned value) must be an inline string literal or an inline array of `Css.rule` values—not a variable reference or a named function reference.

```rescript
// ✅ Valid — inline string literal
module A = %styled.div((~color) => `color: $(color);`)

// ✅ Valid — inline array of Css.rule
module B = %styled.div((~color) => [%css("color: $(color)")])

// ❌ Invalid — can't pass a variable reference
let myCss = "color: red;"
module C = %styled.div(myCss)

// ❌ Invalid — can't pass a named function reference
let fn = (~color) => `color: $(color);`
module D = %styled.div(fn)
```

If you need to build CSS from complex runtime logic, use the [Array API](/reference/array) to compose `Css.rule` values directly.
